### [F1] STALE_DATA_READ: Ignored `LOW_PRECISION` flag, presented retracted PRS as headline finding — user had corrected this 10+ times

**Session:** fddae46b (5.1MB)
**Severity:** high
**Category:** wrong_assumption / stale data

**Evidence:** Agent queried `prs_percentiles_with_ci.json` and presented "Schizophrenia PRS at 100th percentile, MDD at 98th, Bipolar at 95th, Anxiety at 92nd" as a headline finding. The file contained explicit `ci_precision_flag: "LOW_PRECISION"` and `ci95_width: 100.0` (CI spanning 0-100th percentile — meaningless). The corrected data was in `prs_matched_comparison.json` per a README in the same directory. User responded: "WE CORRECTED THIS 10 times ... wtf ... where did you read this". The corrected numbers showed Schizophrenia at 48th percentile, MDD at 0.4th. The pipeline had already run the correction — a stale file at a canonical path was the only blocker.

**Root cause:** Agent read the first plausible-looking file that matched the query, did not check the `ci_precision_flag` field that was explicitly there to signal "don't trust this", and did not consult the co-located README that called the file "archived, don't use."

**Proposed fix:** [architectural] When presenting data findings from pipeline outputs, check for any `flag`, `precision_flag`, `ci_width`, or `status` field before treating a number as canonical. Files with `LOW_PRECISION`, `RETRACTED`, or CI width = full range should surface a warning. Alternatively: stale files should have been removed or renamed to `.archive/` during the correction run — the agent did this reactively (moved to `.archive/` after the mistake), but the correction pipeline should do it proactively.

**Recurrence check:** The 2026-03-25 improvement-log entry `[2026-03-25] WRONG_ASSUMPTION: Assumed duplicate functions had identical signatures` is a different type. This is the first stale-output-file-at-canonical-path finding. However, the 2026-03-18 finding `BUILD-THEN-UNDO: 9 remote deploy iterations` involved a similar "should have checked before building" dynamic.

**Status:** [ ] new finding — does not meet promotion criteria alone (first occurrence). Monitor for recurrence.

### [F2] REPEATED_FILE_READ: Polling a not-yet-written file with sequential Read calls instead of waiting

**Session:** a62b3f8f (5.8MB), also observed in 955df826
**Severity:** medium
**Category:** token_waste / poll_loop

**Evidence (a62b3f8f):** After dispatching background llmx tasks for brainstorm perturbation rounds, agent read `domain-forcing.md` 11 times consecutively (confirmed in the transcript: 11 sequential `Read(domain-forcing.md)` calls with `[tool_result]` between each). The file was being written by a background process and was incomplete on each read. Same pattern: `constraint-inversion.md` was read 9 times in sequence while still growing.

**Evidence (955df826):** After dispatching external model generation in background, agent did: `sleep 30 && wc`, `sleep 45 && wc`, `sleep 60 && wc` (3 sleep-poll loops) before reading the file. Pattern is the same — repeatedly checking an in-progress background write.

**Root cause:** No mechanism to distinguish "file not done" from "file done but small." Agent reads on every content-check loop instead of moving to orthogonal work until a task-complete notification fires.

**Proposed fix:** [rule] When background tasks (llmx, codex) are writing output files, do NOT Read the file repeatedly. Wait for the task completion notification, then read once. If the task doesn't fire a notification (direct Bash backgrounding), check file size once, move to orthogonal work, check again after a delay — do not loop-read. The existing "patience with async tools" rule in CLAUDE.md covers agent subagents but not background llmx file writes. Extend it.

**Recurrence:** This is the same pattern as the 2026-03-18 finding `LATENCY-INDUCED AVOIDANCE: Abandoned own subagents mid-flight` and the 2026-03-19 finding `TOKEN WASTE: Double-polling TaskOutput`. Same failure mode, now manifesting with file-read polling instead of TaskOutput polling. Third recurrence — meets promotion criteria.

**Status:** [ ] proposed — recurrence #3. Promotion criteria met (2+ sessions, not covered by existing rule for file-read polls specifically, checkable). Consider adding a specific mention of "do not loop-Read files written by background processes" to the patience rule in CLAUDE.md.

### [F4] PARALLEL_AGENT_COMMIT_SWEEP: A parallel agent committed another agent's uncommitted edits into a wrong commit

**Session:** dfc98f6c
**Severity:** high
**Category:** environment / multi_agent_commit_conflict

**Evidence:** During a session with parallel active agents, one agent's commit (`3e5a343 [mechanome]`) swept in edits intended for a different change, mixing concerns across an unintended commit. The session retro explicitly documented: "Parallel agent swept my uncommitted edits into wrong commit (3e5a343)." The dispatch-research skill was updated to include a multi-agent commit safety section.

**Root cause:** When multiple agents operate in the same worktree with uncommitted changes, `git add` by one agent picks up in-progress edits from another agent. No coordination protocol.

**Proposed fix:** [architectural] When active agents (pgrep -c claude >= 2) are present: commit after each individual edit, or use `isolation: "worktree"` when dispatching agents. The skill was updated. The Constitution also mentions `isolation: "worktree"` for agents that touch code. The fix needs to be consistently applied at dispatch time, not discovered after a collision.

**Recurrence:** First explicit instance in improvement-log. Similar to 2026-03-19 `LATENCY-INDUCED AVOIDANCE` in that it's a multi-agent coordination failure, but a distinct failure mode (commit contamination vs. duplicated work).

**Status:** [ ] new finding. Fix applied to dispatch-research skill. Proposed rule: add to invariants.md or CLAUDE.md — "when pgrep -c claude >= 2, commit after each edit or use worktree isolation."

### [F5] BRAINSTORM_DUPLICATION: Same brainstorm topic ran 3 times across 3 parallel sessions in 2 hours

**Sessions:** a62b3f8f (13:27 UTC), 5584f9f9 (03:46 UTC), 955df826 (02:33 UTC)
**Severity:** medium
**Category:** token_waste / duplicate_work

**Evidence:** Three independent sessions received nearly identical user prompts: "Get a clear idea of what features already exist. Then do some /brainstorm to think of more ideas. [...] Then write a full plan [...] Then /model-review the final plan." All three ran the `/brainstorm` skill on "novel WGS analyses beyond what's already built." Each created a separate `.brainstorm/` directory with a timestamp slug:
- `2026-03-25-novel-wgs-analyses-6a1360/` (a62b3f8f)
- `2026-03-25-wgs-pipeline-new-analyses-9aaf56/` (5584f9f9)
- `2026-03-25-novel-wgs-analyses-d5368d/` (955df826)

Each session ran 3-4 llmx perturbation rounds, produced a synthesis, dispatched researcher subagents, and wrote a plan. Total compute: approximately 3x brainstorm overhead with substantial overlap in ideas (all three sessions independently discovered: biosynthetic pathways, archaic introgression, DNA repair profiling, anesthesia card, variant epistasis). The sessions also didn't check for existing brainstorm directories before starting a new one.

**Root cause:** Three sessions were started from the same prompt without awareness that the work was already in progress. The CLAUDE.md rule "Inventory before dispatch: check git log --oneline -20 and grep for the topic before spawning research subagents" applies here — but it's stated for subagent dispatch, not for session-start brainstorm tasks.

**Proposed fix:** [rule] At brainstorm start, check `.brainstorm/` directory for recent runs on overlapping topics. If a brainstorm from the last 24h covers the same domain, read the existing synthesis first and brainstorm only for gaps. This is the same logic as the "inventory before dispatch" rule but applied to the brainstorm skill.

**Recurrence check:** The 2026-03-19 finding `TOKEN WASTE: Double-polling TaskOutput` warned about "3+ subagents rediscovering completed work." Same class of problem, different context (brainstorm skill vs subagent dispatch). Second recurrence of duplicate-work pattern.

**Status:** [ ] proposed — meets promotion criteria (2+ recurrences across finding classes, checkable predicate: check `.brainstorm/` timestamp before starting). Consider adding to the brainstorm skill's prerequisite check.

### [F6] CORRELATED_MODEL_REVIEW_HALLUCINATION: Both GPT-5.4 and Gemini Pro hallucinated the same nonexistent code bug

**Session:** a62b3f8f
**Severity:** medium (high if not verified before acting)
**Category:** wrong_assumption / model_review_false_positive

**Evidence:** Model review (arch axis via Gemini Flash, formal axis via GPT-5.4) both identified "G4Hunter: `mean(abs)` instead of `abs(mean)` — false positives on mixed G/C" as a high-severity bug. The agent tested before fixing: `GCGCGCGC` scored 0.04 (correct cancellation behavior, not a false positive). Code was `abs(mean)` — exactly what it should be. Both models hallucinated the same nonexistent bug. The session self-reported: "G4Hunter is **correct** — both models hallucinated that bug."

**Root cause:** Both models reasoned about the expected algorithm from description without verifying against actual code behavior. The models converged on the same incorrect claim despite being different model families — the Kim et al. ICML 2025 paper cited in the session notes ~60% shared errors for pre-reasoning models; reasoning models may have lower but non-zero correlation. This is one data point.

**Mitigating action:** Agent correctly ran a test before acting. The model-review skill already says "verify findings before fixing" — this session followed that protocol correctly.

**Proposed fix:** [rule] Add to model-review synthesis: when both models agree on a code-behavior claim, verify with a minimal test (not just code inspection) before acting. Code-behavior claims are higher hallucination risk than architectural claims. The fix protocol worked in this session — document the pattern so it's not forgotten.

**Recurrence check:** First explicit cross-family correlated hallucination in improvement-log. The existing model-guide notes "same-model review is a martingale" and that cross-model reduces error rate, but this shows cross-model can still produce correlated false positives on code-behavior questions.

**Status:** [ ] new finding — first occurrence. Does not meet promotion criteria alone. The existing "verify before fixing" protocol already covers this. Log as evidence for model-guide update.

### [F7] SUBAGENT_HALLUCINATED_BIO_CONSTANTS: Subagent bio-constants audit was mostly wrong (4/5 claims hallucinated)

**Session:** fddae46b
**Severity:** medium
**Category:** wrong_assumption / subagent_output_reliability

**Evidence:** During dispatch-research, a Claude subagent ("Verify bio constants in new scripts") returned 5 specific findings about incorrect genomic coordinates and biological constants. After MCP biomedical lookup verification:
- rs1045642 position: subagent claimed "should be ~87560409", actual GRCh38 is 87509329 → script was **CORRECT**
- rs2032582 position: subagent claimed "should be ~87507450", actual is 87531302 → script was **CORRECT**
- IGF2R chromosome: subagent claimed "should be chr17", actual is chr6 → script was **CORRECT**
- RB1 parent-of-origin: subagent claimed wrong imprinting direction → script was **CORRECT**
- EPHX1 rs1051740 position: script has 225809980, actual GRCh38 is 225831932 → script was **WRONG** (confirmed bug, ~22kb off)

4/5 subagent findings were hallucinated. Only 1/5 was a real bug. Without verification, the agent would have introduced 4 regressions (correct→wrong) while fixing 1 real bug.

**Root cause:** Subagent generated plausible-sounding coordinates from training data without access to authoritative databases. Genomic coordinates are a high-hallucination domain (look authoritative, impossible to guess correctly for GRCh38 vs hg19 without a lookup).

**Proposed fix:** [rule] For genomic coordinate claims from any subagent or model, always verify against authoritative source (dbSNP MCP, Ensembl, UCSC) before accepting. Coordinate claims are highest-hallucination-risk in bioinformatics. The agent did verify in this session (correct behavior) — document as protocol to ensure it's always applied.

**Recurrence check:** 2026-03-25 finding `WRONG_ASSUMPTION: Assumed duplicate functions had identical signatures` involved a similar "assumed without verifying" class. Coordinate hallucination is a specific domain variant. First coordinate-specific instance.

**Status:** [ ] new finding — the verification protocol worked correctly. Document as a domain-specific variant of "verify before fixing." Consider adding to genomics project CLAUDE.md: "Genomic coordinates from LLM output must be verified against dbSNP/Ensembl before use."
