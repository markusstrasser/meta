### 1. [CONTRADICTION] DuckDB DESCRIBE vs schema_cache.json

**Pattern:** Database Schema Discovery (VERIFIED)
**Frequency:** 262 DESCRIBE calls in intel in 7 days; fires despite schema_cache.json existing with 325-table schema
**Projects:** intel

**Root cause:** CLAUDE.md simultaneously says: (a) "Read `.claude/schema_cache.json` for full 325-table schema" and (b) "DESCRIBE before first query on ANY table." Rule (b) overrides (a) every session. The schema cache exists precisely to eliminate runtime DESCRIBE calls. The rules contradict each other.

**Current cost:** 262 unnecessary DuckDB round-trips per week. Each one is a tool call that could be eliminated by reading the already-loaded schema cache.

**Approaches considered:**
1. **[Selected] Delete the DESCRIBE rule; replace with "trust schema_cache.json; DESCRIBE only if table absent from cache or cache mtime >24h"** — removes the contradiction at source. Zero maintenance.
2. Add a PostToolUse hook blocking DESCRIBE when cache is fresh — more surgical but adds hook complexity for the same result achievable via rule change.
3. Auto-regenerate schema_cache.json on session start — adds latency, no benefit if cache is already fresh.

**Implementation:**
```
# In intel/CLAUDE.md, replace:
"DESCRIBE before first query on ANY table."
# With:
"Read schema_cache.json before querying. DESCRIBE only if a table is absent from the cache."
```

**Blast radius:** intel only
**Reversibility:** easy (one-line CLAUDE.md change)
**Status:** NEW
**Priority:** CRITICAL — 262 wasted queries/week eliminated with 5 minutes of work

### 2. [HOOK] Inline Python3 -c Enforcement

**Pattern:** Inline Python JSON Parsing (VERIFIED CROSS-PROJECT — highest frequency pattern)
**Frequency:** 503 sessions across all 4 projects in 7 days
**Projects:** meta, intel, selve, genomics

**Evidence (verbatim):**
```
Bash: python3 -c "\nimport json\nwith open('indexed/healthkit_parsed.json')..."
Bash: python3 << 'PYEOF'\nimport json, os\nbase = ...
Bash: uv run python3 -c "\nimport json\nd = json.load(open('intel.duckdb'))..."
```

**Root cause:** CLAUDE.md has a ">10 lines → write a .py file" rule but zero enforcement. The constitution says "instructions alone = 0% reliable." This is proof.

**Approaches considered:**
1. **[Selected] PreToolUse:Bash hook — detect multi-line python3 -c patterns, inject advisory** — consistent with existing hook pattern, low false-positive risk (one-liners fine, only multi-line matters)
2. Add a JSON query MCP tool so agents don't reach for python — addresses symptom not cause; the pattern covers CSV, subprocess, AST, not just JSON
3. Add a `json-query` just recipe — too narrow; doesn't cover non-JSON uses

**Implementation sketch:**
```bash
# pretool-inline-python-warn.sh (PreToolUse:Bash)
INPUT=$(cat)
TOOL=$(echo "$INPUT" | python3 -c 'import sys,json; print(json.load(sys.stdin).get("tool_name",""))' 2>/dev/null)
[ "$TOOL" != "Bash" ] && exit 0
CMD=$(echo "$INPUT" | python3 -c 'import sys,json; print(json.load(sys.stdin).get("tool_input",{}).get("command",""))' 2>/dev/null)

# Detect multi-line embedded python
LINE_COUNT=$(echo "$CMD" | python3 -c "
import sys
cmd = sys.stdin.read()
# Count newlines in -c argument or heredoc
import re
m = re.search(r\"python3\s+-c\s+'([^']+)'\", cmd, re.DOTALL)
if m:
    print(m.group(1).count('\n'))
else:
    # heredoc
    m2 = re.search(r'<<\s*[A-Z]+\n(.+?)(?=\n[A-Z]+|\Z)', cmd, re.DOTALL)
    print(m2.group(1).count('\n') if m2 else 0)
" 2>/dev/null || echo 0)

if [ "${LINE_COUNT:-0}" -gt 8 ]; then
    echo '{"additionalContext": "STYLE: Multi-line python3 -c detected ('"$LINE_COUNT"' lines). Write to a .py file instead: Write(/tmp/task_name.py) then Bash(uv run python3 /tmp/task_name.py). Inline scripts >10 lines are harder to debug and not reusable."}'
fi
exit 0
```

**Blast radius:** All projects (global hook candidate)
**Reversibility:** easy
**Status:** NEW
**Priority:** HIGH — 503 sessions is the highest raw frequency across all patterns this week

### 3. [RECIPE] AST Syntax Validation — just syntax-check

**Pattern:** AST Syntax Validation (VERIFIED CROSS-PROJECT)
**Frequency:** 170 meta + 55 intel + 69 genomics sessions = 294+ sessions reinventing the same `python3 -c "import ast; ast.parse(...)"` check
**Projects:** meta, intel, selve, genomics

**Evidence (verbatim):**
```
Bash: python3 -c "import ast; ast.parse(open('scripts/modal_premode.py').read()); print('OK')"
Bash: for script in scripts/...; do python3 -c "import ast; ast.parse(open(\"$script\").read())"; done
```

**Root cause:** No shared recipe. Each agent reinvents it every session. The pattern is identical across all projects — the abstraction is obvious but nobody has written it down.

**Approaches considered:**
1. **[Selected] Add `just syntax-check` to each project's justfile** — one definition, referenced forever, integrates with existing just workflow
2. PostToolUse:Write hook that auto-runs ast.parse when .py files are written — too noisy (fires on every Write, not just when validation is needed)
3. Document the one-liner in CLAUDE.md — instructions-only, 0% reliability per constitution

**Implementation sketch:**
```justfile
# Add to meta, intel, genomics, selve justfiles
syntax-check:
    #!/usr/bin/env python3
    import ast, glob, sys, os
    errors = []
    src_dirs = ["scripts", "src", "tools", "substrate"]  # customize per project
    for d in src_dirs:
        if not os.path.isdir(d):
            continue
        for f in glob.glob(f"{d}/**/*.py", recursive=True):
            try:
                ast.parse(open(f).read())
            except SyntaxError as e:
                errors.append(f"{f}:{e.lineno}: {e.msg}")
    if errors:
        print("\n".join(errors)); sys.exit(1)
    print(f"OK")
```

**Blast radius:** All projects (each adopts independently)
**Reversibility:** easy
**Status:** NEW
**Priority:** MEDIUM — eliminates reinvention in 294 sessions/week, 30 min across 4 projects

### 5. [HOOK] llmx Zombie Process Cleanup at Session Start

**Pattern:** llmx Process Contention & Zombie Killing (VERIFIED — intel)
**Frequency:** 10+ kill sequences across 2 intel sessions. The diagnostic + cleanup cycle costs 3-5 turns per occurrence.
**Projects:** intel (model-review dispatches llmx most heavily)

**Evidence (verbatim):**
```
Bash: ps aux | grep llmx | grep -v grep | wc -l  → 8
Bash: pkill -f "llmx chat" 2>/dev/null
Bash: kill -9 43128 56491 43106 56478 2>/dev/null; echo "Killed zombies"
```

**Approaches considered:**
1. **[Selected] SessionStart hook that kills orphaned llmx processes proactively** — prevents zombies from blocking new dispatches, telemetry logged
2. Add `--timeout` flag to all llmx dispatches — prevents zombies from forming; complementary, not exclusive
3. Add `just llmx-clean` recipe — manual, same problem as current (agent must remember to run it)

**Implementation sketch:**
```bash
# sessionstart-llmx-cleanup.sh
ZOMBIE_PROCS=$(pgrep -f "llmx chat" | wc -l | tr -d ' ')
if [ "$ZOMBIE_PROCS" -gt 2 ]; then
    pkill -f "llmx chat" 2>/dev/null || true
    ~/Projects/skills/hooks/hook-trigger-log.sh "llmx-cleanup" "clean" \
        "killed ${ZOMBIE_PROCS} zombie llmx processes at session start" 2>/dev/null || true
    echo '{"additionalContext": "Cleaned '"${ZOMBIE_PROCS}"' zombie llmx processes from prior session."}'
fi
exit 0
```

**Blast radius:** intel (where model-review runs); could be global
**Reversibility:** easy
**Status:** NEW
**Priority:** MEDIUM — eliminates 5-10 wasted turns per model-review session in intel

### 6. [HOOK] codebase-map.py Auto-Refresh on .py File Changes

**Pattern:** Documentation Co-Update (CLAUDE.md + codebase-map.md) (VERIFIED)
**Frequency:** 86+15+19 sessions across meta/genomics/intel with codebase-map.md edits — many are manual regeneration after code changes
**Projects:** meta, genomics, intel

**Evidence (verbatim from genomics):**
```
Edit(/Users/alien/Projects/genomics/CLAUDE.md)
Edit(/Users/alien/Projects/genomics/.claude/rules/codebase-map.md)
Bash: git commit -m "[docs] Update CLAUDE.md and codebase-map.md..."
```

**Root cause:** `codebase-map.py` exists and automates this, but agents run it manually rather than automatically. The second edit (codebase-map.md) is always redundant with what `codebase-map.py` would generate.

**Approaches considered:**
1. **[Selected] PostToolUse hook on Write/Edit to .py files — auto-runs codebase-map.py in background** — eliminates the manual step; runs async so no blocking
2. SessionEnd hook that runs codebase-map.py once — lower overhead but stale during the session
3. `just update-map` recipe — manual, same problem
4. **[Inversion explored]** Make codebase-map.md the source of truth, generate module structure from it — obviously wrong direction but the constraint forced clarifying that freshness matters more than thoroughness

**Implementation sketch:**
```bash
# posttool-codebase-map-refresh.sh (PostToolUse:Write|Edit)
INPUT=$(cat)
FPATH=$(echo "$INPUT" | python3 -c 'import sys,json; d=json.load(sys.stdin); print(d.get("tool_input",{}).get("file_path",""))' 2>/dev/null)
CWD=$(echo "$INPUT" | python3 -c 'import sys,json; print(json.load(sys.stdin).get("cwd",""))' 2>/dev/null)

# Only trigger on .py files in source directories (not tests, not .claude/)
if echo "$FPATH" | grep -qE '\.py$' && ! echo "$FPATH" | grep -qE '\.claude/|tests/|__pycache__'; then
    # Run async — don't block the agent
    uv run python3 ~/Projects/meta/scripts/codebase-map.py "$CWD" --quiet > /dev/null 2>&1 &
fi
exit 0
```

**Blast radius:** meta, genomics, intel (projects with codebase-map.py setup)
**Reversibility:** easy
**Status:** NEW
**Priority:** MEDIUM — automates a manual step that fires across 120 sessions/week

### 7. [HOOK] Search Burst Threshold — MCP vs Web Differentiation

**Pattern:** Search Burst Hook Interference (VERIFIED — genomics, biomedical-mcp)
**Frequency:** 19 genomics sessions explicitly mention "search burst hook"; 2 sessions with explicit complaints the hook is counterproductive for their workflow
**Projects:** genomics, biomedical-mcp

**Evidence (verbatim):**
```
"Search burst hook hit. Let me process what I have, then continue with different tool types."
"The search burst hook is counterproductive for this specific workflow... I'm forced to serialize what should be parallel"
"The hook wants me to process existing results before searching more."
```

**Root cause:** The burst threshold is calibrated for web search exploration (where batching is correct). Research MCP workflows (search_papers, mcp__brave, mcp__exa in parallel across databases) are inherently parallel and benefit from simultaneous dispatch.

**Approaches considered:**
1. **[Selected] Raise threshold for MCP research tools (8) vs web search (4)** — surgical, preserves the hook's core value while reducing false positives in research contexts
2. Add per-session override flag — too much agent autonomy, creates a new class of prompt-injection surface
3. Remove the hook — wrong; it catches real waste in non-research sessions
4. Context-aware threshold (higher during researcher skill) — interesting but hard to implement reliably without knowing when skills are active

**Implementation sketch:**
```bash
# In pretool-search-burst.sh, after detecting tool name:
MCP_RESEARCH_TOOLS="mcp__research|mcp__brave|search_papers|search_preprints|mcp__scite"
if echo "$TOOL_NAME" | grep -qE "$MCP_RESEARCH_TOOLS"; then
    BURST_LIMIT=8   # Research MCP: parallel DB queries are legitimate
else
    BURST_LIMIT=4   # Web search: batch more conservatively
fi
```

**Blast radius:** All projects (global hook modification)
**Reversibility:** easy (threshold change in existing hook)
**Status:** NEW
**Priority:** MEDIUM — reduces friction in research sessions that are currently artificially serialized

### 8. [WILD CARD] Session Registry — The Missing Coordination Layer

**Pattern:** Stop Hook Uncommitted Changes Ceremony (VERIFIED — ALL PROJECTS)
**Frequency:** 100+ session files across all projects contain "Uncommitted changes" — stop hook fires, agent checks git status, sees changes from OTHER active sessions, explains "not mine to commit," exits. One wasted turn every session end across all projects.
**Projects:** meta, intel, selve, genomics

**The deeper pattern:** This same coordination failure shows up in 4 different symptoms:
1. Stop hook fires for other sessions' uncommitted files
2. Memory pressure gate blocks without telling the agent which other sessions are the problem
3. DuckDB lock contention with no visibility into which process holds it
4. llmx zombie processes from prior sessions blocking new dispatches

All four are the same root: **sessions don't know about each other.** Each session acts as if it owns the world when it shares the machine with 3-7 siblings.

**The design that wants to emerge:** A lightweight session registry that tracks:
- Which sessions are running (process ID, project, started_at)
- Which files each session has modified in the current run
- Which resources each session holds (DuckDB write lock, llmx child processes)

The `~/.claude/active-agents.json` file is a partial prototype. It tracks session identity but not file/resource ownership.

**What becomes possible with a session registry:**
- Stop hook filters out other sessions' uncommitted files (without the explanation ceremony)
- Memory pressure hook tells the agent exactly which session is the bottleneck
- DuckDB preflight identifies which session holds the lock
- llmx zombie detection knows which parent session spawned which child process
- Steward can coordinate work across sessions without risking file conflicts

**Approaches considered:**
1. **[Selected for design, not immediate implementation] Extend active-agents.json with per-session modified-files list** — SessionStart hook initializes the entry; PostToolUse:Write/Edit appends to modified-files; SessionEnd hook removes the entry. Consumer hooks read this file before making coordination decisions.
2. Use git log timestamps as a proxy for session file ownership — works for committed files, misses in-progress changes
3. Use file mtime + session start mtime comparison — current approach in stop-uncommitted-warn.sh; partial solution, doesn't scale to arbitrary resource types
4. Accept the coordination cost — 100+ wasted turns/week is low per-instance

**Why this is the wild card:** This proposal challenges the assumption that Claude Code sessions are independent. They're not — they share a machine, a repo, and infrastructure. The entire hook system was designed for single-session operation. A session registry is the architectural change that makes the hook system multi-session-aware.

**Implementation path:**
- Phase 1 (30 min): Add `modified_files` field to active-agents.json via PostToolUse:Write hook
- Phase 2 (1 hr): Update stop-uncommitted-warn.sh to filter using the registry
- Phase 3 (2 hr): Update memory pressure, DuckDB preflight, llmx cleanup to use registry

**Blast radius:** All projects (global infrastructure change)
**Reversibility:** medium (architectural change, but removable)
**Status:** NEW — design proposal, not ready for immediate implementation without Phase 1 validation
**Priority:** HIGH architectural signal, LOW immediate urgency (each symptom has a simpler per-symptom fix)

## System Trajectory

Three themes across 7 days of sessions:

**1. Single-session assumptions in a multi-session world.** The stop hook ceremony, DuckDB lock contention, llmx zombies, and memory pressure fallbacks are all the same underlying problem: sessions don't coordinate. The session registry is the architectural fix; the per-symptom fixes (proposals 4, 5) are valid in the short term.

**2. Reinvented logic as a signal of missing shared primitives.** 503 sessions reinvent python3 -c JSON parsing. 294 sessions reinvent ast.parse checks. These patterns are identical across projects. The agents lack a "personal toolkit" — a set of project-scoped scripts that survive sessions and are referenced rather than reinvented. The `/tmp` pattern is a proto-toolkit; what's missing is persistence across sessions and discoverability.

**3. Proposal accumulation without implementation.** Prior design reviews have 50+ proposals. 0 built (per today's existing 2026-03-21.md). The design review skill generates proposals faster than the system consumes them. This review deliberately keeps proposals to 8 (vs 22+ patterns found) and includes implementation time estimates on each.

**The 80/20:** Proposal 1 (DuckDB DESCRIBE contradiction) eliminates 262 wasted queries/week with a one-line CLAUDE.md fix. Five minutes. Do this one first.
