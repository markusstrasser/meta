# 2026-03-17: Shared knowledge substrate with domain profiles, not one unified stack

## Context
Three repos now exhibit the same structural problem in different forms: understanding compounds more slowly than documents. The repeated work is not primarily new research; it is stale path/count/index repair, manual correction propagation, and machine state leaking into prose.

Observed local evidence:

- `intel`'s own architecture memo says the repo has a state-model problem: beliefs, evidence, and outcomes are split across markdown, trackers, priors, and ledgers instead of flowing through one canonical schema. See `intel/analysis/research/intelligence-engine-knowledge-accrual-memo-2026-03-05.md`.
- `selve` has strong memo contracts, append-only guards, and domain validators, but its epistemics audit explicitly says citation existence does not guarantee claim correctness and that missing verification/disconfirmation artifacts make regressions harder to catch. See `selve/docs/research/MEMO_CONTRACT.md`, `selve/docs/reports/research_epistemics_audit_2026_02_19.md`, and `selve/scripts/verify_variant_claims.py`.
- `genomics` is not just another memo repo. It is a pipeline-first system with explicit evidence thresholds, benchmark gates, classification calibration, and a stated boundary that genomics produces outputs while `selve` consumes them for phenotype cross-checks. See `genomics/CLAUDE.md`.
- Meta's current architecture already assumes: centralized orchestration, fail-open hooks, shared global skills/hooks with intentional project divergence, and domain-branched epistemic checks rather than one universal semantic regime. See `meta/CLAUDE.md` and `meta/research/epistemic-architecture-v3.md`.

Git-history signal informing this decision:

- `intel` and `selve` both show high churn in `CLAUDE.md` and other doc/index surfaces from stale counts, stale paths, and recency fixes rather than purely new understanding.
- `selve` has positive proof that correction propagation is valuable (`PRS` retraction propagated across 15+ docs), but that propagation is still a special operation rather than a general substrate capability.
- `genomics` shows a different but related pattern: benchmark-gated pipeline and curation changes, explicit calibration harnesses, and recurring truth-surface alignment fixes after upstream evidence changes.

**Tension with prior infrastructure assessment:** `meta/research/cross-project-infra-factoring.md` (2026-03-06) concluded that "the projects are more different than similar at the system level" and killed a shared utility library plan. That conclusion stands for utility code — `load_jsonl`, `parse_ts`, `safe_float` are trivially regenerated by agents and the domain-specific logic IS the system. But provenance state is a different category: it's a cross-cutting concern (like embeddings, which *did* extract successfully into `emb`) with a domain-agnostic core (assertion depends on evidence, derivation links inputs to outputs) and domain-specific overlays. The pilot phase (§ Migration order) tests whether provenance actually meets the `emb` bar — domain-agnostic capability, clean interface, real deduplication value — or whether it turns out to be another candidate that "doesn't extract" per the infra-factoring assessment.

This decision covers shared infrastructure design only. It does **not** authorize cross-repo implementation work or shared hook/skill deployment without explicit approval, per meta's constitution.

## Alternatives considered
1. **Fully unified stack: one prompt, one skill set, one hook set, one schema, same semantics everywhere.**
   Pros: maximal standardization, simplest story.
   Cons: wrong fit for the repos as they exist. `intel` prediction resolution, `selve` memo verification/disconfirmation, and `genomics` ACMG-style classification are materially different state machines. This would turn current doc churn into schema churn.

2. **Per-project independent stacks.**
   Pros: best local fit, low coordination cost.
   Cons: duplicates tooling and loses the ability to share correction infrastructure, generated indexes, identifier conventions, and propagation logic. Cross-project lessons remain trapped in repo-local scaffolding.

3. **Native graph/hypergraph-first implementation.**
   Pros: dependency traversal and many-hop lineage queries are natural graph problems; hypergraph semantics match multi-premise derivations.
   Cons: adds infrastructure and tooling risk before proving that current pain is graph-query limited. Mainstream agent workflows, git auditing, markdown generation, and local CLI/DB tooling are still easier on relational tables. Native hypergraph tooling exists (`XGI`, `HyperNetX`, `HyperGraphDB`), but it is not the operational center of the current stack. Neo4j-style modeling guidance still recommends reified intermediate nodes for n-ary relations.

4. **Generated indexes + propagation scripts only.**
   Pros: lowest build cost (~200 lines per repo). Solves stale counts, stale paths, and index repair — the most visible current pain — without new infrastructure.
   Cons: generated indexes solve the *symptom* (stale docs) but not the *cause* (no canonical dependency registration). Manual correction propagation (e.g., selve's PRS retraction across 15+ docs) cannot be automated by index regeneration alone — it requires knowing which downstream assertions depend on the retracted claim. Without dependency edges, propagation remains a manual, error-prone, special operation each time. If the pilot shows that the actual pain is 90% stale indexes and 10% correction propagation, this option should be reconsidered.

5. **Shared knowledge substrate with domain profiles.**
   Shared canonical state and dependency registration, with project/domain-specific validators, statuses, and workflows layered on top.
   Pros: captures shared structure without forcing a fake common ontology. Fits the current global/shared/project layering already documented in meta.
   Cons: requires discipline about what belongs in the shared kernel vs project overlays.

## Decision
Adopt **Option 5**: one shared knowledge substrate, with **domain profiles** for `intel`, `selve`, and `genomics`.

The shared kernel is intentionally narrower than the first-pass design. The cross-model critique surfaced a valid overbuild risk around workflow semantics. Therefore:

### Shared kernel
The canonical shared objects are:

- `Assertion`
- `Evidence`
- `Artifact`
- `Derivation`
- typed `Relation`

`Event` is kept as an append-only audit/change log generated from writes, not as a first-class domain object that all repos manipulate directly.

`Task` stays in the orchestrator / project workflows, not in the shared knowledge substrate.

This keeps the shared layer focused on **provenance state and dependency structure**, not generic workflow orchestration.

### Dependency model
Do **not** use a native hypergraph database initially.

Model multi-premise dependencies through **reified derivation nodes**:

- inputs (`Assertion`, `Evidence`, `Artifact`) point to a `Derivation`
- the `Derivation` points to downstream `Assertion` or `Artifact`

This provides hypergraph semantics without adopting a native hypergraph stack.

### Canonicality
Markdown remains the narrative/audit layer. It is not the source of truth for machine state.

Hand-maintained index/count/path docs should be replaced by generated inventories and views backed by the shared substrate where possible.

### Propagation
Propagation is deterministic and edge-driven:

1. a write, validation result, or external correction emits a change-log record
2. dependency traversal finds affected downstream nodes
3. downstream state is marked `stale`, `needs_review`, `invalid`, or `superseded`
4. project-local workflows decide what to rebuild, refresh, or escalate

Hooks may trigger registration and validators, but hooks are **not** the propagation engine.

### Registration enforcement

Dependency-capture completeness is the binding constraint (flagged by Gemini 3.1 Pro critique). The substrate is only as useful as registration is complete. Concrete mechanisms:

1. **Write-path hooks** on canonical doc directories (analogous to selve's `docs-guard.sh`). When an agent writes/modifies a canonical doc, the hook checks: are the assertions in this doc registered in the substrate? If not, warn (advisory). Promote to blocking only after false-positive measurement, per constitutional P3.
2. **Registration-first write flow.** The `knowledge-accrual` skill/tool should register assertions and dependencies first, then generate the markdown surface. Registration-before-prose inverts the current write flow (prose first, registration never).
3. **Completeness audit script.** Periodic scan (daily or per-session): diff canonical docs against registered assertions, surface unregistered claims. Backstop for cases where hooks don't fire (direct file edits, non-agent sessions).
4. **Incremental adoption.** Don't require 100% registration coverage at launch. Track coverage % per project. The substrate is useful for propagation at ~60%+ coverage; below that, it's an index with gaps. Coverage % is the leading indicator of whether the substrate is being adopted or ignored.

### Shared vs project-specific setup
Do **not** unify prompt/hook/skill behavior into one mega-stack.

Use a layered model:

- global constitution + global mechanical hooks
- shared tool layer and shared skill shells
- project/domain overlays
- task-local prompt/context

This is consistent with meta's current documented architecture: shared skills/hooks, project rules diverge intentionally, and domain-specific epistemic checks branch by domain.

## Per-project profile consequences
### Intel
Intel uses the shared substrate as a prediction- and thesis-adjacent profile.

- shared kernel stores assertions, evidence, artifacts, derivations, relations
- profile adds prediction/resolution/outbox semantics on top
- important states include forecast class, falsifier, resolution status, and postmortem linkage
- propagation targets include theses, priors, watchlists, and outbox proposals

Intel is the best first candidate for a pilot because it already has partial canonical stores (`intel_ledger.py`, `prediction_tracker.py`) and a live need for correction propagation.

### Selve
Selve uses the shared substrate as a memo/entity/protocol profile.

- shared kernel stores assertions, evidence, artifacts, derivations, relations
- profile adds memo contract, verification log, disconfirmation log, and source-grade expectations
- important states include `verified`, `unverified`, `inference`, `contradicted`
- propagation targets include research memos, entity pages, and `active_protocol.md`

Selve is the second pilot because it already has the strongest memo discipline and the clearest examples of manual correction propagation.

### Genomics
Genomics uses the shared substrate as a pipeline/output profile, not a memo-first profile.

- shared kernel stores assertions, evidence, artifacts, derivations, relations
- profile adds benchmark-gated tool integration, calibrated thresholds, and ACMG-/ClinVar-style status semantics
- important subjects include variant, gene, phenotype, sample, and drug
- important states include `pathogenic`, `likely_pathogenic`, `VUS`, `likely_benign`, `benign`, plus `research_only` and benchmark-gated computational evidence
- propagation targets include curation outputs, review packets, dashboards, and downstream exports consumed by `selve`

Genomics should **not** be forced into selve's memo workflow. Its primary contract is pipeline outputs with explicit calibration and provenance, then phenotype/context interpretation happens downstream.

## Tool / hook / prompt design implications
### Shared tools
Create one shared tool layer for:

- registering assertions/evidence/artifacts/relations
- querying impact / lineage
- rebuilding generated indexes/views
- validating identifier references and dependency registration

### Shared skills
Use shared skill shells, not one shared behavior blob:

- `researcher` remains the shared research shell
- add a shared `knowledge-accrual` or equivalent shell for registration / propagation-aware workflows
- project overlays constrain what counts as a valid assertion, what evidence hierarchy applies, and which validators run

### Hooks
Keep hooks thin and mostly deterministic:

- enforce IDs / registration on canonical docs
- run local validators
- prevent obvious destructive/stale writes
- trigger registration or rebuild jobs

Do **not** make hooks responsible for semantic contradiction detection or full propagation.

### Prompts
Do **not** create one unified mega-prompt for all three repos.

Use layered prompts:

1. global constitutional rules
2. project charter / constitution
3. domain profile
4. task-local context

## Implementation shape
The shared kernel should start relational, local, and inspectable.

### Storage location

One SQLite database per project, shared schema. Not one central DB.

Rationale: each project already has its own SQLite patterns (intel: `intel_ledger.py`/DuckDB, meta: `orchestrator.db`, selve: agent coordination). A shared central DB creates a cross-project runtime dependency — any project's agent session would need the shared DB accessible, and failures cascade. Per-project DBs with the same schema keep projects independently functional while enabling cross-project queries via a read-only federation tool (MCP or script that opens multiple DBs).

Cross-project dependency edges (§ below) are stored as foreign references (`project:id`) rather than direct foreign keys — the target may not be locally available.

### Cross-project edges

The existing genomics→selve bridge (`phenome_behavior_bridge.py` reads selve's `interpreted/` via `SELVE_ROOT`) is the one live cross-project dependency. If genomics reclassifies a variant (e.g., VUS→likely_benign), selve's memos citing that classification should be marked stale.

Cross-project propagation uses the same mechanism as within-project (change-log → traversal → stale marking), but with a lightweight bridge:

1. Each project's substrate emits change-log records for its own objects.
2. A cross-project propagation script (in meta, cron or orchestrator) reads change logs from all projects, resolves `project:id` references, and marks affected downstream nodes stale in the target project's DB.
3. The target project's local workflows handle the rebuild.

This is deliberately minimal — one script reading multiple DBs, not a distributed event system. The pilot should prove that within-project propagation works before wiring cross-project edges.

### Recommended first implementation shape

- SQLite with shared schema (one DB per project)
- relational tables for shared kernel objects and typed relations
- JSON payload fields only where the domain overlay genuinely differs
- materialized dependency views and recursive traversal queries where needed
- generated markdown/index surfaces for git-visible auditability (mark with `<!-- GENERATED -->` header to prevent agent edits)

Graph export is acceptable later for analysis, but a native graph or hypergraph database is **not** the default starting point.

## Migration order
1. **Pilot in `intel`**
   Prove one end-to-end path: registered assertions/evidence -> dependency capture -> stale marking -> refreshed generated views.
2. **Add `selve`**
   Reuse shared kernel and registration rules; add memo/disconfirmation/verification overlay.
3. **Integrate `genomics` later**
   Only after the kernel is stable in `intel` + `selve`. Genomics has stricter evidence gates and output contracts; it should not be the proving ground for a first-pass substrate.

External evidence that informed specific design choices:

- W3C PROV data model (`Entity` → `Activity` → `Entity` triad): validated that provenance is modelable as a small set of node types with typed edges. Our kernel maps: `Assertion`/`Artifact` ≈ PROV `Entity`, `Derivation` ≈ PROV `Activity`, `Evidence` ≈ PROV `Entity` with a source role. We don't adopt PROV's `Agent` (agents are in orchestrator, not the substrate).
  https://www.w3.org/TR/prov-dm/
- Nanopublications (assertion + provenance + publication info): the structural separation of "what is claimed" from "why we believe it" from "where it was published" maps directly to our Assertion/Evidence/Artifact split. Confirms the kernel isn't arbitrary.
  https://nanopub.net/introduction
- Neo4j n-ary modeling guidance: confirms reified intermediate nodes (our `Derivation`) over native hyperedges for n-ary relations in practical systems.
  https://neo4j.com/docs/getting-started/data-modeling/modeling-designs/

Cross-model critique run on 2026-03-17 via `llmx`:

- **Gemini 3.1 Pro** objected that the first-pass design was overbuilt and argued that eventual consistency / dependency-capture completeness are the real operational risks. Its strongest surviving point: propagation is only as good as dependency registration completeness.
- **GPT-5.4** objected that the shared substrate could become a fake common ontology and recommended trimming the shared kernel to provenance state rather than workflow. That critique changed the decision: `Task` stays out of the shared substrate, and `Event` is treated as a generated audit log rather than a generic first-class workflow object.

## Revisit if
- shared kernel changes begin causing schema-version churn across repos
- recursive dependency traversal becomes a dominant operational bottleneck
- many-hop interactive impact analysis becomes a routine workflow, not an occasional debugging task
- `intel` and `selve` fail to show a stable end-to-end pilot path
- `genomics` proves to need a materially different substrate rather than a profile overlay
- native graph tooling becomes operationally cheaper than materialized relational dependency views in the current stack
- registration coverage stalls below 50% after 30 days — the substrate is not being adopted and the infra-factoring assessment was right: this doesn't extract either
- the pilot reveals that >80% of the pain is stale indexes solvable by Option 4 (generated indexes only) without dependency tracking
- per-project DBs create schema drift that a single shared DB would have prevented
