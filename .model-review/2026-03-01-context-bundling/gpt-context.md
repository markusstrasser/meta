# CONTEXT: Cross-Model Review of Context Bundling Infrastructure Design

# PROJECT CONSTITUTION
Quantify alignment gaps. For each principle, assess: coverage (0-100%), consistency, testable violations.

## Key Principles
1. Architecture over instructions (enforce with hooks/tests/scaffolding)
2. Recurring patterns become architecture (10+ uses → hook/skill/scaffolding)
3. Cross-model review for non-trivial decisions
4. Research is first-class (divergent → convergent → dogfood → analyze)
5. Measure before enforcing
6. Self-modification by reversibility + blast radius

## Goals
- Maximize autonomous agent capability, measured by declining supervision
- Primary metric: ratio of autonomous-to-supervised work
- Resource constraints: single human operator, cost-conscious, local Mac + cloud APIs

# PROBLEM

Cross-model review (sending codebase context to Gemini + GPT for adversarial critique) currently:
- Takes 3-10 minutes per review
- Serial context assembly (Claude reads files one-by-one, writes context bundles)
- Serial model dispatch (two `llmx` calls run sequentially)
- No caching (same codebase re-read from scratch each time)
- No structured output (free text → lossy prose synthesis)
- Burns Claude's own tokens on copy-paste work

# PROPOSED DESIGN

## 1. Pre-computed Context Views

Each project gets `.context/` (gitignored) with repomix-generated XML outputs:

| View | Content | Token estimate |
|------|---------|---------------|
| full.xml | Everything | 50-200K |
| src.xml | Source code only | 20-100K |
| docs.xml | Documentation only | 5-30K |
| infra.xml | Config/CI/hooks | 5-20K |
| signatures.xml | Tree-sitter function signatures (~70% smaller) | 10-40K |
| filetree.txt | Directory structure only | 1-3K |
| diffs.xml | Working tree changes | 1-10K |
| constitution.md | Stable prefix (constitution + GOALS + arch ref) | 2-5K |

Generated by a `context-gen` script using per-view repomix config files.

## 2. Generation Triggers

- **SessionEnd hook** — auto-refresh after each work session
- Manual `context-gen` invocation
- Before any model-review

## 3. Prompt Caching via Stable Prefix

| Provider | Caching | Discount | Requirement |
|----------|---------|----------|-------------|
| Gemini | Implicit auto | 90% | Byte-identical prefix |
| OpenAI/GPT | Automatic >1K tokens | 50% | Byte-identical prefix |

Structure: `[stable prefix: constitution + base context] + [variable: topic + prompt]`

## 4. Parallel Dispatch

Claude Code fires two Bash tool calls in one message (already supported):
```
Bash 1: cat composed.md | llmx -m gemini-3.1-pro-preview ... > gemini-out.md
Bash 2: cat composed.md | llmx -m gpt-5.2 --stream ... > gpt-out.md
```

## 5. Structured Output

Request JSON findings arrays instead of free text. Synthesis becomes array merge, not prose summarization.

## 6. llmx Changes

| Change | Effort | Why |
|--------|--------|-----|
| Fix `--json` to set `response_format` on API call | Small | Currently only affects stderr logging, not model output |
| Add `--system` flag to CLI | Small | System message → better cache separation |
| Add `-f/--file` flag | Small | Cleaner than `cat \| llmx` piping |

## 7. What We're NOT Building

- `llmx review` subcommand (shell `&` + `wait` is simpler)
- Automatic view selection (Claude picks from manifest)
- Python repomix wrapper (CLI is fine)

# CURRENT llmx ARCHITECTURE

- CLI (`cli.py`) → `providers.chat()` → `litellm.completion()`
- No custom provider abstraction — LiteLLM handles HTTP/auth/response normalization
- CLI has NO `--system` flag — stdin goes into user message
- Python API (`api.py`) HAS system message support via `LLM.chat(prompt, system=...)`
- `--json` flag only affects stderr logging format, NOT `response_format` parameter
- `--compare` already does parallel ThreadPoolExecutor dispatch but sends same context to all models
- `batch()` function exists in API with configurable parallelism
- In-process cache decorator exists but not wired into chat by default

# KEY CONSTRAINTS

1. **Single human operator** — automation must be reliable enough to not need babysitting
2. **Cost-conscious** — cache hits save 50-90% on input tokens
3. **Repomix is already installed** (v1.11.1) and used regularly
4. **llmx is a local project** (`~/Projects/llmx/`) — we can modify it freely
5. **Both models are thinking models** — temperature locked at 1.0, reasoning_effort is the control knob
6. **Gemini gets large context (50-200K), GPT gets focused context (10-40K)** — different strengths

# QUANTITATIVE QUESTIONS

1. **ROI of SessionEnd generation:** If average session changes <5% of files, regenerating all views is wasteful. What's the breakeven? Should we only regenerate views whose included files changed?
2. **Cache hit rate estimate:** With stable prefix, what % of reviews would get cache hits? Depends on review frequency and cache TTL.
3. **Cost comparison:** Current cost per review (no caching, manual assembly) vs proposed (pre-computed views, caching). Estimate in $/review.
4. **Structured output quality tradeoff:** JSON schema constrains model output. Does the synthesis quality gain from easier merging outweigh the loss of creative/nuanced prose?
5. **manifest.json necessity:** Is tracking token counts and timestamps worth the complexity, or is the filesystem (mtime) sufficient?
6. **Number of views:** 8 views × N projects = many files. Is this the right granularity, or should views be generated on-demand?
7. **The `--system` flag:** Both providers cache user message prefixes too. Is the system/user split actually needed for caching, or is it just cleaner separation of concerns?
