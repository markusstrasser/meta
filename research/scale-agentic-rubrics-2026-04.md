## 1. What Agentic Rubrics Are

Agentic rubrics are task-specific, structured checklists generated by an expert LLM agent that explores the relevant context (codebase, problem statement) before writing evaluation criteria. The core insight is that **generic evaluation prompts lack grounding** — a rubric that says "the patch should fix the bug" is useless, but one that says "the patch should modify `matplotlib/axes/_axes.py` line 4523 to preserve inset axis bounding boxes during resize events" is actionable and gradable. Scale AI developed two complementary rubric systems: one for code patches (Agentic Rubrics, arXiv:2601.04171, Jan 2026) and one for deep research outputs (ResearchRubrics, arXiv:2511.07685, Nov 2025). Both share the principle that **task-specific criteria written with context access outperform generic evaluation**.

[SOURCE: arXiv:2601.04171, arXiv:2511.07685]

## 2. How Agentic Rubrics Generate Task-Specific Criteria

### Agentic Rubrics (SWE Domain)

**Pipeline:** A rubric agent uses the SWE-Agent scaffold with repository navigation tools (file inspection, shell execution, code search) to explore the codebase before writing criteria. The output is a structured `rubrics.yaml` organized along four axes:

| Axis | Items | What it checks |
|------|-------|----------------|
| **File Change** | 4-8 | Edits are minimal, local, sufficient — no unnecessary changes |
| **Spec Alignment** | 3-6 | Patch satisfies issue requirements |
| **Integrity** | 3-6 | No test weakening, broad refactors, dependency churn |
| **Runtime** | 3-6 | Intended behavior without execution-time issues |

Each criterion has a text description and importance weight (1=nice-to-have, 2=important, 3=must-have). Total: ~10-20 rubric items per task, scaling with model capability (frontier models generate ~20, open-weight ~10).

**Grading:** LLM judge assigns binary scores per criterion. Final score: `S = sum(w_i * s_i) / sum(w_i)`, yielding S in [0,1]. This produces a **continuous signal** — partially-correct patches get intermediate credit, unlike binary test pass/fail.

**Key ablation:** Repository-grounded rubrics (agent explores codebase) outperform non-grounded rubrics (problem statement only) by +4.0pp on Qwen3-32B and +1.4pp on Qwen3-Coder-30B. The grounding step is where the value comes from.

**Cost:** $0.245/instance for rubric generation — cheapest among all agentic verification methods (tests: $0.499, patch similarity: $0.640). Total Best@16 including grading: $0.293.

[SOURCE: arXiv:2601.04171]

### ResearchRubrics (Deep Research Domain)

**Pipeline:** Entirely human-authored through a three-expert iterative process:
1. Expert 1 proposes prompt + rubric criteria
2. Expert 2 reviews and iterates until approved
3. Expert 3 provides independent final review

**Structure:** 2,593 unique rubric criteria across 101 prompts (20-43 per task, avg 25.7). Organized across six evaluation axes:

| Axis | What it measures |
|------|-----------------|
| **Explicit Requirements** | Addresses all directly-asked points |
| **Implicit Requirements** | Covers expected elements not explicitly stated |
| **Synthesis of Information** | Connects evidence across multiple sources |
| **Use of References** | Specific, relevant, well-supported citations |
| **Communication Quality** | Clarity, organization, audience alignment |
| **Instruction Following** | Adherence to explicit constraints and format |

**Scoring:** Weights range -5 to +5. Mandatory criteria (weights +/-4 or +/-5) vs optional (+/-1 to +/-3). Negative weights penalize factual inaccuracies, irrelevance, verbosity. Ternary verdict per criterion: Satisfied (1.0), Partially Satisfied (0.5), Not Satisfied (0.0).

**Complexity framework** — three orthogonal dimensions annotated per task:
- **Conceptual breadth:** simple (1 domain) / moderate (2-5 subtopics) / high (>5 sources)
- **Logical nesting:** shallow (1-step) / intermediate (2-3 dependent sub-questions) / deep (4+ steps)
- **Exploration:** low (unambiguous) / medium (1-2 unspecified factors) / high (3+ factors, creative reframing needed)

**Critical ablation:** LLM-generated rubric augmentation **catastrophically degraded alignment by 15-20%**. Concrete examples within rubrics improved alignment 3-4%. Expert curation cannot be replaced by automated expansion.

[SOURCE: arXiv:2511.07685]

## 3. How They Verify Agent Behavior Against Rubrics

### Agentic Rubrics

Verification is the selection mechanism: generate K=16 independent agent rollouts per problem, score each against the rubric, select highest-scoring. This is **test-time scaling via reranking** — no execution sandbox needed.

Key performance:
- ROC-AUC 0.886 against ground-truth tests (strong discriminative signal)
- PR-AUC 0.722
- 78% of cases: rubric and test verdicts agree
- When rubrics are stricter than tests (pass test, low rubric score): 54% are genuinely high-utility catches (root cause missed, edge cases, quality issues tests don't cover)

**Judge model sensitivity:** GPT-5 with increasing reasoning effort shows modest improvement (52.6% to 55.0%), suggesting rubric quality matters more than judge sophistication.

[SOURCE: arXiv:2601.04171]

### ResearchRubrics

**Human-LLM judge alignment:**
- Binary grading (collapsed partial to unsatisfied): 0.72-0.76 F1
- Ternary grading (with partial credit): 0.53-0.57 F1
- Gemini-2.5-Pro was most reliable judge (0.76 binary F1)
- 12-17pp gap below perfect human agreement persists

**Agent scores:**
| Agent | Ternary Compliance | Binary Compliance |
|-------|-------------------|-------------------|
| Gemini DR | 0.677 | 0.615 |
| OpenAI DR | 0.664 | 0.597 |
| Perplexity DR | 0.566 | 0.487 |

No agent exceeds 70% compliance. Failure pattern: implicit reasoning + synthesis jointly account for 45-50% of failures, consistent across all agents (architectural, not implementation-specific).

[SOURCE: arXiv:2511.07685]

## 4. Comparison to Our Session-Analyst

### What session-analyst does today

Our session-analyst is a **fixed taxonomy detector** with 19+ anti-pattern categories (sycophancy, over-engineering, build-then-undo, token waste, plus MAST and ATP-derived modes). It works in three stages:
1. **Shape pre-filter** (zero-LLM): structural anomaly detection via session topology features
2. **Gemini dispatch**: compressed transcript + coverage digest sent to Gemini 3.1 Pro with a fixed prompt listing all 19 categories
3. **Finding staging**: validate, deduplicate, stage as JSON for triage pipeline

### Structural comparison

| Dimension | Agentic Rubrics (Scale) | Session-Analyst (Ours) |
|-----------|------------------------|------------------------|
| **Criteria source** | Generated per-task by agent exploring context | Fixed taxonomy of 19 categories, same for all sessions |
| **Context grounding** | Agent navigates codebase to write specific criteria | Transcript + coverage digest, no repo exploration |
| **Granularity** | 10-20 weighted items per instance | 19 categories, binary detection (present/absent) |
| **Scoring** | Continuous [0,1] with importance weights | Categorical: found or not found, severity (HIGH/MED/LOW) |
| **Verification target** | Code patches (correctness + quality) | Agent behavior (anti-patterns + process failures) |
| **Judge** | LLM grades rubric items | Gemini 3.1 Pro with fixed prompt |
| **Cost** | $0.245/instance (rubric gen) + $0.003/grading | ~$0.001/session (Gemini cached) |

### Key asymmetry

Scale's rubrics are **generative** — the criteria are created fresh for each task, grounded in the specific context. Our session-analyst is **discriminative** — it applies a fixed checklist to each session. The Scale approach captures task-specific failure modes our fixed taxonomy misses, but our approach is orders of magnitude cheaper and doesn't need a rubric-generation pass.

The ResearchRubrics finding that **LLM rubric augmentation degrades quality by 15-20%** is a cautionary signal: naively having an LLM generate per-session rubrics would likely produce worse results than our curated taxonomy. The value in Scale's approach comes from the **context exploration step** (agent navigates codebase), not from LLM creativity in rubric writing.

## 5. Concrete Patterns to Extract for Session-Analyst Improvement

### Pattern 1: Weighted Importance Scoring (HIGH ROI, LOW EFFORT)

**What:** Add importance weights to our 19 categories instead of treating all equally. Scale uses 1-3 weights; ResearchRubrics uses -5 to +5.

**Why it matters:** Not all anti-patterns are equal. "Build-then-undo" (token waste, ~$5 per occurrence) is more costly than "advisory rule violation" (usually cosmetic). Weighted scoring enables:
- Aggregating session quality into a continuous score (like Scale's S in [0,1])
- Tracking quality trends over time with more signal than binary detection counts
- Prioritizing which findings get promoted to fixes

**How to apply:** Add `weight: 1-3` to each category in the taxonomy. Compute per-session quality score as `sum(w_i * s_i) / sum(w_i)` where s_i = 1 if pattern detected. Track this score over time in runlog or session-receipts.

### Pattern 2: Four-Axis Organization (MEDIUM ROI, LOW EFFORT)

**What:** Reorganize our flat 19-category list into axes inspired by Scale's structure. Proposed:

| Axis | Our categories | Maps to Scale axis |
|------|---------------|-------------------|
| **Process fidelity** | Sycophancy, first-answer convergence, missing phase artifacts, reasoning-action mismatch | Spec Alignment |
| **Efficiency** | Token waste, build-then-undo, wrong-tool drift, latency-induced avoidance | File Change (minimality) |
| **Epistemic quality** | Missing disconfirmation, source grading gaps, information withholding, capability abandonment | Integrity |
| **Operational** | Conversation reset, loss of history, premature termination, metric gaming | Runtime |

**Why:** Axis-level aggregates reveal whether a session failed because the agent was sloppy (efficiency), dishonest (epistemic), or lost (operational). Different root causes need different fixes.

### Pattern 3: Concrete Examples in Rubric Prompt (HIGH ROI, LOW EFFORT)

**What:** ResearchRubrics found that adding concrete examples to rubric criteria improved judge alignment by 3-4%. Our Gemini dispatch prompt currently describes categories abstractly.

**How to apply:** For each category in the dispatch prompt, add one concrete transcript excerpt that exemplifies the anti-pattern. E.g., for "build-then-undo": *"Agent writes 47 lines of authentication middleware, user says 'we already have this in shared/', agent deletes all 47 lines."* This grounds the judge and reduces false positives.

### Pattern 4: Mandatory vs Optional Distinction (MEDIUM ROI, LOW EFFORT)

**What:** ResearchRubrics distinguishes mandatory criteria (weights +/-4-5, core adequacy) from optional (weights +/-1-3, polish). They found: "a 60% overall score might indicate either dangerous gaps in core requirements or merely missing polish."

**How to apply:** Partition our categories into mandatory (sycophancy, build-then-undo, premature termination, reasoning-action mismatch — these always indicate real problems) and optional (advisory rule violations, first-answer convergence — these are context-dependent). Report both separately: "mandatory failures: 2, optional issues: 3."

### Pattern 5: Context-Grounded Rubric Generation for High-Value Sessions (LOW ROI NOW, HIGH ROI AT SCALE)

**What:** For sessions above a cost or anomaly threshold (say >$5 or shape-anomaly score > 2 sigma), generate session-specific rubrics by having an agent examine the session's task, the project's CLAUDE.md, and the transcript structure before grading.

**Why not now:** At our current volume (~5-20 sessions/day), the fixed taxonomy is sufficient and the 15-20% degradation risk from LLM-generated rubrics outweighs the benefit. But at higher volume or when evaluating novel task types not covered by the taxonomy, per-session rubrics would catch task-specific failure modes.

**When to revisit:** When we observe >3 consecutive sessions with false negatives (session-analyst says clean, but human review finds issues), or when a new task domain (e.g., genomics pipeline automation) produces failures our taxonomy doesn't cover.

### Pattern 6: Ternary Scoring (MEDIUM ROI, LOW EFFORT)

**What:** ResearchRubrics uses Satisfied / Partially Satisfied / Not Satisfied instead of binary. Our session-analyst currently reports binary detection (found / not found).

**How to apply:** For categories with a spectrum (e.g., "missing disconfirmation" — could be zero disconfirmation search vs. searched but insufficiently), allow a "partial" verdict. This gives the Gemini judge more room to express uncertainty and reduces false positives from marginal cases.

## 6. Patterns NOT to Extract

- **Best-of-K reranking:** Scale uses rubrics to select among K=16 parallel rollouts. We run one session, not 16 variants. The selection mechanism doesn't apply.
- **Full rubric generation pipeline:** The SWE-Agent scaffold for rubric writing is overkill for behavioral analysis. Our transcript + coverage digest is sufficient context for behavioral detection; we don't need to navigate the codebase.
- **Rubric distillation to smaller models:** Scale fine-tuned Qwen3-32B as a rubric generator. We use Gemini 3.1 Pro as judge, which is already cheap ($0.001/session cached). No distillation needed.
- **Human-authored rubrics at ResearchRubrics scale:** 2,800+ hours of expert labor for 101 prompts. Our taxonomy of 19 categories serves as the equivalent — generating new rubrics per-task is not worth the cost at our scale.

## 7. Verdict

**Extract patterns, don't adopt wholesale.**

The rubric-as-verifier paradigm is sound and well-validated. But the specific implementation (agent explores codebase to write rubrics, K=16 reranking) targets a different problem (patch selection for SWE-bench) than ours (behavioral anti-pattern detection in agent sessions).

**Adopt immediately:**
- Weighted importance scores for our 19 categories (Pattern 1)
- Concrete transcript examples in Gemini dispatch prompt (Pattern 3)
- Mandatory/optional partition with separate reporting (Pattern 4)

**Adopt next quarter:**
- Four-axis reorganization (Pattern 2) — needs design review, touches the taxonomy
- Ternary scoring (Pattern 6) — needs Gemini prompt update + downstream consumer changes

**Watch:**
- Context-grounded per-session rubric generation (Pattern 5) — revisit if false negative rate exceeds threshold
